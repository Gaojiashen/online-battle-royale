# 结算不再等所有人

## 问题

在当前的裁定流程（`行动框架_裁定流程.md`）中，游戏以"轮"为单位推进——法官在截止时间收集所有玩家的行动，统一处理，统一反馈。

这隐含了一个假设：**每一轮必须等所有玩家都提交行动后才能推进。**

但这个假设有问题：

- 迟交的玩家拖慢全体进度
- 某玩家是否还在游戏内，影响其他人的体验
- 世界的时间尺度被人为的"轮"割裂，而非自然地向前流动

## 灵感核心

**游戏按现实时间（天）向前推动，而不是按"所有玩家都已行动"推动。**

但一天之内，玩家的多个行动应该在"玩家提交 → 法官收到 → 法官立即结算"这条链路上尽可能快地完成，不必等同一个"轮次窗口"关闭。

### 具体设想

```
时间轴（现实时间）
─────────────────────────────>
  Day 1         Day 2         Day 3
  │             │             │
  玩家A 提交 ─→ 法官立即结算 ─→ 反馈
  玩家B 提交 ─→ 法官立即结算 ─→ 反馈
  玩家A 再提交 ─→ 法官立即结算 ─→ 反馈（同一天内可多次交互）
  世界事件 ──→ 自动触发 ──→ 玩家感知
```

### 关键原则

1. **不强制等待**：玩家 A 提交行动后，只要不涉及与玩家 B 的冲突/互动，法官可以直接结算并反馈，不需要等玩家 B。
2. **按天推进，而非按轮推进**：每天是一个自然的时间窗口。一天结束时，世界状态推进一次（势力行动、NPC 行动、环境变化等）。
3. **同一天内可多次行动**：玩家一天内可以提交多个行动，每个行动独立结算——不需要凑满"3 个行动位然后等轮次结束"。
4. **冲突行动**：如果玩家 A 的行动涉及玩家 B（如攻击），而 B 尚未提交当天行动，则此行动进入"待定"状态，在 B 提交后或日切时按规则裁定。

## 为什么这是对的

- **法官体验**：不必攒一批再处理，随到随结，工作流更平滑
- **玩家体验**：提交后很快得到反馈，沉浸感更强，"等待"感减少
- **世界感**：世界是活的，不是一帧一帧跳动的幻灯片
- **异步参与**：不同节奏的玩家都能玩——一天提交一次可以，一天提交十次也可以

## 对现有设计的冲击

| 领域 | 影响 |
|------|------|
| 行动位 | 可能不再需要"每轮 3 行动位"的硬限制，改为"每天行动次数上限" |
| 裁定流程 | 从"批量处理"变为"流式处理"，法官工具需要支持在线待办队列 |
| 冲突判定 | 需要明确"冲突中的两个行动谁先谁后"的规则（按提交时间？按声明优先级？） |
| 世界推进 | 势力/NPC 自动行动从"每轮一次"变为"每天日切时触发" |
| 截止时间 | 不再是"周五晚 22:00 截止提交"，而是"每天 00:00 日切，日切前的行动即时结算" |

## 开放问题

- 一天的行动次数上限是多少？还是用体力/精力值替代？
- 日切时刻是否固定（如每天 00:00）？还是由法官灵活决定？
- 冲突待定队列如何可视化给法官？
- 是否需要"行动槽位"的概念——每个玩家同时只能有 N 个待结算行动？
